## 2. 개선 기획 — 우선순위별

### P0 — 실사용 정합성 (디자인 파트너 받기 전 필수)

**P0-1. 연차 잔액 단일 소스화**
- 현상: 연차 계산이 3곳에 분산 — `calc_annual_leave()` 실계산, `leave_balances`(이월), 화면별 used 집계 쿼리(타입 포함 범위가 화면마다 다름: 어떤 곳은 sick 포함, 어떤 곳은 annual만).
- 문제: 직원이 보는 잔여연차와 HR이 보는 값, 연차촉진 대상 산정값이 어긋날 수 있음. 연차촉진(§61)은 법적 증빙이라 특히 위험.
- 방향: `get_leave_balance(user_id, year)` 단일 헬퍼로 통합 (발생–사용–이월을 한 곳에서), 모든 화면·촉진·퇴직정산(미사용수당)이 이 헬퍼만 사용. 반차(0.5일) 차감 일관성 포함.

**P0-2. 급여 "자동계산 → 담당자 확정" 2단계화** (saas_plan §3 명시 항목)
- 현상: 급여 생성 즉시 확정·명세 발송 훅까지 작동. 미리보기 모달은 있으나 생성 후 정정 플로우가 없음.
- 방향: payslips에 `status`(draft/confirmed) 추가 → 생성=draft(직원 비공개) → HR 검토 화면에서 개별 수정 → 일괄 확정 시 공개+이메일 발송. "세무사 검증 전 정확성 보장 표방 금지" 문구와 짝을 이루는 구조.

**P0-3. 데모/시드 데이터 정리**
- 현상: ①직원 1번(employee@company.com)의 manager_id가 employee 역할 사용자를 가리킴 — 승인 체인 데모가 어색함 ②`tenant_None.db` 생성 버그 흔적(원인 미조사) ③UI 언어 혼재(아래 P1-3)와 겹쳐 데모 품질 저하.
- 방향: 시드에서 부서별 매니저 role 정합성 보장, tenant_None 원인 조사(get_tenant_db_path 호출부에 tenant_id None 가드).

**P0-4. 회귀 방지 자동화**
- 현상: 이번 점검으로 잡은 버그 6건 중 4건이 "존재하지 않는 컬럼/엔드포인트/속성" 유형 — 실행 전 잡을 수 있는 것들.
- 방향: ①이번에 만든 3역할 라우트 스윕을 `tests/sweep.py`로 저장소에 편입, 릴리즈 워크플로우 6단계에 "스윕 통과" 추가 ②템플릿 `url_for` 전수 렌더 스모크.

### P1 — HR 담당자 실무 효율 (제품의 팔리는 이유 강화)

**P1-1. "오늘 할 일" 승인 허브 (대시보드 인박스 확장)** ★ 페이지 연결 관점 핵심
- 현상: admin 인박스가 휴가/증명서/발령/퇴직 4종만 집계. Phase C에서 생긴 **목표 승인 대기, 등급 이의신청, 입사 예정 D-day, OT 승인**은 각자 페이지에 흩어져 있고 알림으로만 연결됨. HR 1인 운영 타깃과 어긋남.
- 방향: 대시보드 인박스에 신규 4종 편입 + "결재 대기함" 단일 화면(모든 pending 문서를 한 테이블에서 유형 필터로) 신설. 매니저 대시보드에도 팀원 목표 승인·이의신청 위젯.

**P1-2. 결재선 확장**
- 현상: v1.2.2에서 휴가/증명서/인사발령 3종만 설정화. OT·경비(benefit_claims)는 고정.
- 방향: 동일한 `approval_chains` 프레임으로 OT(매니저 전결 ↔ 2단계), 복리후생 환급 편입. 장기적으로 금액 조건부 결재선(예: 발령 중 급여 변경 ≥10%는 무조건 HR).

**P1-3. UI 언어 일관화 (한글 우선)**
- 현상: 국내 컴플라이언스 SaaS 포지셔닝인데 직원 등록 폼(Full Name/Employment Info), 일부 대시보드 문구, 버튼이 영어. 화면마다 혼재.
- 방향: 템플릿 전수 스윕으로 실무 화면 전부 한글화(법령 용어는 병기). 특히 **직원이 보는 화면**(프로필/근태/급여명세) 우선.

**P1-4. CSV 임포트 왕복 완성**
- 현상: 들어오는 길(직원/입사예정자 임포트)은 완성. 나가는 길은 Excel 내보내기뿐 — 수정 후 재업로드 불가.
- 방향: "내보내기 → 수정 → 재업로드로 일괄 수정(급여·부서·매니저)" 왕복 플로우. 엑셀로 일하던 회사가 이관하는 실제 경로.

### P2 — 직원 사용성·완성도

**P2-1. 직원 모바일 3대 동선 최적화**: 출퇴근 체크인, 휴가 신청, 급여명세 확인이 직원 사용의 90% — 이 3개 화면만이라도 모바일 전용 레이아웃 검증(현재 반응형 CSS는 있으나 체크인 버튼 도달 거리·탭 크기 미점검).

**P2-2. 성과 매니저 뷰 팀원 필터**: 부서 전체 나열 → 팀원 수 10명+ 시 검색/미완료만 보기 필터.

**P2-3. 알림 정리**: 카테고리 일관성(perf/performance 혼용 → 'perf' 통일), 알림 목록에서 읽음 일괄 처리.

**P2-4. 온보딩 투어 확장** (기존 보류 항목): Phase C 신기능(성과 단계 스텝퍼, 입사 예정자, 결재선)에 data-tour 부여.

### 페이지 간 유기적 연결 — 종합 평가

**잘 연결된 것 (Phase C에서 개선됨)**: 성과 주기 스텝퍼→캘리브레이션→이의신청→Merit 인상, 입사예정자→직원 등록 프리필→온보딩 자동 가동, 채용 합격→직원 전환, 캘리브레이션→보상 검토.

**끊겨 있는 것 (P1-1이 해소)**: 승인 업무가 페이지별로 분산(대시보드 인박스 미편입 4종), 알림을 놓치면 발견 경로가 없음.

**어긋났던 것 (수정 완료)**: 다면평가 관리→캘리브레이션 오링크, 오퍼 GET 500, 결제 플로우 깨진 링크.

## 3. 권장 실행 순서

1. **개선 스프린트 1 (P0)**: P0-1 연차 단일화 → P0-2 급여 2단계 → P0-4 스윕 자동화 → P0-3 시드 정리
2. **개선 스프린트 2 (P1)**: P1-1 승인 허브 → P1-3 한글화 스윕 → P1-2 결재선 확장 → P1-4 CSV 왕복
3. 이후 **Phase D** (PostgreSQL 전환, Effective Dating, 알림 센터 통합, 연말정산 — saas_plan §6)

> P0가 끝나면 "디자인 파트너 1~2개사에 실데이터 3개월" (saas_plan §1 목표)을 시작할 수 있는 상태로 판단.
